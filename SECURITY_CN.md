## 报告漏洞

> [!IMPORTANT]
> 私人安全披露计划已暂停，恢复时间另行通知。请将包含修复的补丁以公开 PR 的形式直接提交到仓库。邮件将被忽略。

如果你发现了本项目中属于[涵盖范围](#涵盖范围)的安全漏洞，请私下报告。**不要将其作为公开 issue 披露。** 这样我们可以在公开曝光之前与你合作修复问题，降低漏洞在补丁发布前被利用的风险。

请通过私密的 [安全公告](https://github.com/ggml-org/llama.cpp/security/advisories/new) 进行披露。

本项目由志愿者团队以尽力而为的方式维护。因此，请在公开曝光前至少给我们 **90 天**的时间来修复问题。

### 要求

提交报告前，请确保满足以下要求：

- 你已阅读并完全理解本策略。
- AI 仅允许作为辅助手段使用，如 [AGENTS.md](AGENTS.md) 所述。我们不接受完全由 AI 编写的报告。
- 你的报告必须包含一个可运行的概念验证（Proof-of-Concept），形式为脚本和/或附件。

如果未满足这些要求，维护者保留关闭报告的权利。

### 涵盖范围

只有属于以下项目部分的漏洞才被视为有效。对于不在列表中的问题，请作为 issue 报告。

- `src/**/*`
- `ggml/**/*`
- `gguf-py/**/*`
- `tools/server/*`，**不包括**以下主题：
  - Web UI
  - 标记为实验性的功能
  - 不建议在不受信环境中使用的功能（例如 router、MCP）
  - 可能导致拒绝服务攻击（DoS）的 bug

请注意，[安全使用 llama.cpp](#安全使用-llamacpp) 下的所有主题均不被视为 LLaMA C++ 的漏洞。

对于 `vendor` 目录中的漏洞，请直接向第三方项目报告。

## 安全使用 llama.cpp

### 不受信的模型

运行不受信的模型时请格外小心。此类别包括由未知开发者创建的模型或使用来自未知来源数据训练的模型。

*请始终在安全的隔离环境中运行不受信的模型，例如沙箱*（如容器、虚拟机）。这有助于保护你的系统免受潜在恶意代码的侵害。

> [!NOTE]
> 模型的可信度不是非黑即白的。你必须根据具体模型及其与你的用例和风险承受能力的匹配程度，始终确定适当的安全等级。

### 不受信的输入

一些模型接受各种输入格式（文本、图像、音频等）。转换这些输入的库具有不同的安全级别，因此隔离模型并仔细预处理输入以降低脚本注入风险至关重要。

在处理不受信的输入时，为实现最大安全性，你可能需要采取以下措施：

- **沙箱化**：隔离推理运行的环境。
- **预分析**：检查模型在面对提示注入时的默认表现（例如使用[提示注入模糊测试](https://github.com/FonduAI/awesome-prompt-injection?tab=readme-ov-file#tools)）。这将为你在后续主题中需要投入多少精力提供参考。
- **更新**：保持 LLaMA C++ 和你的库始终更新到最新安全补丁。
- **输入净化**：在将数据输入模型之前，严格净化输入。涉及的技术包括：
  - **验证**：对允许的字符和数据类型实施严格规则。
  - **过滤**：移除潜在的恶意脚本或代码片段。
  - **编码**：将特殊字符转换为安全表示。
  - **检测**：运行能识别潜在脚本注入的工具（例如[检测提示注入尝试的模型](https://python.langchain.com/docs/guides/safety/hugging_face_prompt_injection)）。

### 数据隐私

为保护敏感数据免受潜在泄露或未授权访问，对模型执行进行沙箱化至关重要。这意味着在安全的隔离环境中运行模型，有助于缓解多种攻击途径。

### 不受信的环境或网络

如果你无法在安全隔离的环境中运行模型，或者模型必须暴露在不受信的网络中，请确保采取以下安全预防措施：

- 不要使用 RPC 后端、[rpc-server](https://github.com/ggml-org/llama.cpp/tree/master/tools/rpc) 和 [llama-server](https://github.com/ggml-org/llama.cpp/tree/master/tools/server) 功能（参见 [PR #13061](https://github.com/ggml-org/llama.cpp/pull/13061)）。
- 确认下载的任何文件（如预训练模型权重）的哈希值与已知的正确值匹配。
- 如果通过网络发送数据，请进行加密。

### 多租户环境

如果你打算在共享内存中并行运行多个模型，你有责任确保模型之间不会相互交互或访问彼此的数据。主要关注点包括：租户隔离、资源分配、模型共享和硬件攻击。

1. **租户隔离**：模型应使用强隔离方法分开运行，以防止未授权的数据访问。网络隔离对于隔离至关重要，它可以防止对数据或模型的未授权访问，以及恶意用户以其他租户身份发送计算图执行。

2. **资源分配**：一个模型导致的拒绝服务可能影响整个系统的健康状况。应实施速率限制、访问控制和健康监控等保护措施。

3. **模型共享**：在多租户模型共享设计中，租户和用户必须理解运行他人提供的代码所带来的安全风险。由于没有可靠的方法来检测恶意模型，建议对模型执行进行沙箱化以降低风险。

4. **硬件攻击**：GPU 或 TPU 也可能受到攻击。[研究](https://scholar.google.com/scholar?q=gpu+side+channel)表明，对 GPU 的侧信道攻击是可能的，这可能导致同一系统上同时运行的其他模型或进程的数据泄露。
